# DC-10.2 — Repair Engine Wiring

**Gate:** DC-10.2 Runtime Stabilization  
**Purpose:** Wire the repair engine into the desktop runtime in a read-only, architecture-preserving way.

---

## Entry Point

**`electron/desktop/engine-backed-provider.ts`**

- Invokes `runBoundedRepairLoop()` with seeded demo input (ANALYZE, sessionId `dc102_demo_1`)
- Maps `RepairRunOutcome` + verdict + handoff → backbone types → DC-3 read models
- Caches outcome per process for consistency across views

---

## Engine-Powered Views

| View | Source |

| RepairRunView | Engine outcome → backbone mapper |
| StateMachineView | Engine outcome → backbone mapper |
| FailureTimelineView | Engine outcome → backbone mapper |
| RepairStrategyView | Verdict + handoff → backbone mapper |

## IPC Handler Flow

`electron/ipc/handlers/backbone-read.ts`:

- `GET_REPAIR_RUN_VIEW`, `GET_STATE_MACHINE_VIEW`, `GET_FAILURE_TIMELINE_VIEW`, `GET_REPAIR_STRATEGY_VIEW`, `GET_DECISION_VIEW` → engine-backed provider
- `GET_GPT_ANALYSIS_VIEW`, `GET_TELEMETRY_VIEW` → mock backbone service

---

## Safety

- Read-only: No write, mutate, execute, or merge APIs
- Human authority: Outcome terminates at HUMAN; JulesPlaceholder returns frozen
- No UI imports into engine; no engine imports into renderer components
